## 一、AI 基础设施构建：本地 Ollama 的能力边界被严重低估

### 发生了什么

项目最终运行在用户本地 Windows 机器上：Ollama + Qwen3.5:35b-a3b（MoE 模型），上下文窗口 16,384 tokens，推测 32GB RAM。写作脚本在这套环境中遭遇 600 秒超时——系统提示吃掉了 10,822 tokens（66% 的上下文），只剩 ~638 tokens 给模型生成，模型在这种窒息状态下反复重试直到超时。

### 根因分析

**核心错误**：开发阶段没有做 token 预算计算。Skill 文件从 v1.0 到 v4.0 持续膨胀（从几百字到 13,528 字），每次修改都在"增加规则"，没有人算过"这些规则加起来占多少 token"。直到用户实际运行报错，才发现系统提示已经比可用上下文还大。

**次要错误**：
- `think=True` 默认开启——Qwen 的思考模式在 critique 和 compare 步骤各追加 ~12 分钟延迟，用户感知的总时间可能超过 40 分钟
- `num_ctx=16384` 对于 35B MoE 模型在消费级硬件上已经是上限，但代码中没有任何防护
- 没有在脚本启动时输出 token 预算诊断信息，出了问题只能盲猜

### 可复制的经验

1. **Token 预算必须是第一性设计约束**。在为任何本地模型写 prompt 之前，先画一张预算表：

   ```
   总上下文 = num_ctx
   系统提示 ≤ 总上下文 × 10%（理想）/ 20%（上限）
   参考材料 = 总上下文 × 40-60%
   生成空间 ≥ 总上下文 × 20-30%
   ```

   如果系统提示超过 20%，说明 Skill 设计有问题——要么内容应该移到检索阶段动态注入，要么需要做精简。

2. **本地推理的硬件画像必须提前收集**。至少需要知道：RAM、VRAM、模型量化方式、实际可用 num_ctx。这些信息决定了整个工程的设计空间。本项目中，直到出问题才知道用户的硬件配置，导致所有设计都是在"假设资源充足"的前提下进行的。

3. **脚本启动时必须打印 token 预算诊断**。最终修复中加入的 `[Token 预算]` 输出应该从 v1.0 就存在。格式：

   ```
   [Token 预算] skill=800 输出=1600 开销=800 剩余=13184 (总=16384)
   [参考分配] 2篇 x 3000字/篇 (可用=16480字)
   ```

4. **thinking 模式是奢侈品，不是默认选项**。在资源受限环境下，`think=True` 应该只用于最终的一次性评估，而不是每个中间步骤。

## 二、技术栈选择：从 7 条路径到最终落地的偏差

### 发生了什么

Day 1 就产出了 `TRAINING_GUIDE.md`，清晰列出 7 条技术路径（System Prompt → RAG → Long-context → LoRA → Full Fine-tune → OpenAI Fine-tune → Multi-Agent），并推荐"RAG + LoRA"为最优组合。但最终实际落地的方案是：

**System Prompt + 本地关键词检索 + 单模型串行工作流**

这比原始推荐方案低了至少两个档次。没有向量数据库（用的是 `jieba` 关键词匹配），没有 embedding，没有 RAG 框架，没有微调。

### 为什么偏差这么大

1. **用户环境约束**：用户在 Windows 本地运行 Ollama，没有 GPU 做微调，没有向量数据库基础设施，也不使用云 API。这直接排除了路径 2/4/5/6。
2. **交互方式约束**：用户希望在 Open WebUI 中使用 Skill 文件，这意味着最终方案必须兼容"纯提示词"场景，Python 脚本是增强而非唯一入口。
3. **时间约束**：4 天内从零到可用，没有时间搭建 ChromaDB + embedding pipeline。
4. **渐进式滑坡**：项目没有在 Day 1 就锁定"我们选路径 X"，而是边做边加功能，从 Skill 文件 → 加分类 → 加脚本 → 加三稿 → 加去AI化，每一步都合理，但累积起来变成了一个复杂度远超原始路径 1 的系统，却没有路径 2 的核心优势（语义检索）。

### 可复制的经验

1. **在 Day 1 就锁定技术路径，并写下"选择理由"和"放弃理由"**。本项目应该在 TRAINING_GUIDE.md 写完后，立即追加一节"本项目选择路径 1+7 的变体，原因是……"，这样后续所有决策都有锚点。

2. **技术路径选择的决策树应该以"运行环境"为根节点**，而不是以"效果上限"为根节点：

   ```
   有 GPU？ → 路径 4 (LoRA)
   有云 API 预算？ → 路径 6 (OpenAI FT) 或路径 2 (RAG)
   纯本地 + 无 GPU？ → 路径 1 (Prompt) + 路径 7 的简化版
   ```

3. **关键词检索 vs 向量检索的差距在"作家风格"场景中可能没有想象中大**。和菜头的文章主题鲜明，标题信息密度高，jieba 分词 + TF-IDF 匹配在这个特定语料上可能已经足够。如果要升级，优先升级到 BM25，而不是直接跳到向量数据库。

## 三、人工对话引导介入的时机：哪些介入是关键的，哪些是多余的

### 关键介入点（高 ROI）

| 时机 | 用户介入内容 | 影响 |
|------|-------------|------|
| Day 1 开头 | "我要爬和菜头的博客" | 定义了整个项目方向 |
| Day 1 中段 | 确认爬虫已完成 → 转向"接下来怎么用" | 推动了从数据采集到应用构建的跃迁 |
| Day 3 | 反馈"输出质量很差，score 全 0" + 粘贴日志 | 触发了核心 bug 的发现（token 溢出） |
| Day 3 | 反馈"松鼠尾巴反复出现在推理链" | 触发了第二轮修复（标题污染问题） |
| Day 3 | git fetch 失败 + 提供本地 git log | 让 AI 确认了 main 分支状态不一致 |

### 低效介入点（低 ROI 或可优化）

| 时机 | 发生了什么 | 改进方案 |
|------|-----------|----------|
| 多次 PR 链接确认 | 用户反复说"PR 链接错误"，AI 反复检查 | AI 应在首次创建 PR 后立即验证链接有效性 |
| Day 1 技术路径讨论 | 产出了详细的 7 路径对比文档 | 如果 Day 1 就确认了用户环境，这个文档可以缩减为 2-3 段 |
| 中间多次"帮我把文件推到仓库" | 网络问题 + PR 状态混乱导致反复操作 | AI 应在首次推送失败后主动提供 zip 下载替代方案 |

### 可复制的经验

1. **用户反馈实际运行结果是最高价值的介入**。本项目中，"score 全 0"这条反馈的信息密度远高于任何设计讨论。因此应该尽早让用户运行一个最小可行版本（哪怕是 `--quick` 模式），用实际输出驱动迭代。

2. **AI 应该主动收集环境信息，而不是等到出问题**。理想流程：

   ```
   AI: "请运行以下命令并贴出结果：
     1. ollama list
     2. 系统内存大小
     3. 是否有独立显卡"
   ```
   
   这三条信息在 Day 1 就能收集，能避免后续 80% 的 token 预算问题。

3. **"帮我推到仓库"类的机械操作应该有标准化流程**。AI 不应该每次都从头判断 git 状态，而是有一个固定的检查清单：fetch → check diff → squash → push → verify → create PR → verify PR URL。本项目在这个流程上反复出错，浪费了大量对话轮次。

## 四、讨论主题的必要性：三稿迭代是否过度设计

### 争议点

Skill 文件定义了一个雄心勃勃的工作流：三稿并行 → 自检 → 去AI化 → 比较打分 → 迭代优化（最多 3 轮，最多 9 篇草稿）。这个设计在理论上无可挑剔——多样性生成 + 自我批评 + 迭代优化是当前 AI 写作的最佳实践。

但在实际运行中：
- 本地 Qwen 35B 生成一篇 2000 字文章需要 3-5 分钟
- 三稿 = 9-15 分钟（仅生成，不含评分）
- 加上 critique × 3 + compare + 可能的迭代 = **标准模式预计 30-45 分钟**
- 用户的 `--quick` 模式（单稿）已经是 3-5 分钟

**核心问题**：对于一个"模仿特定作家风格"的任务，三稿之间的差异是否足够大到值得多花 10 倍时间？

### 分析

在本地小模型（35B MoE，实际激活参数可能只有 3-5B）上，三稿的差异主要来自温度采样的随机性，而不是真正不同的"角度"和"论证路径"。模型对"切入角度不同"这种高级指令的理解能力有限，三稿可能只是同一篇文章的三个略有不同的改写。

但三稿机制本身不是问题，问题是**在当前硬件条件下的 ROI**：

| 模式 | 时间 | 质量提升预期 | ROI |
|------|------|-------------|-----|
| quick (1稿) | 3-5 min | 基线 | 高 |
| standard (3稿×1轮) | 15-20 min | +10-20% | 中 |
| standard (3稿×3轮) | 45-60 min | +15-25% | 低 |

### 可复制的经验

1. **工作流复杂度应该与模型能力匹配**。如果模型是 GPT-4o 或 Claude 3.5（强指令遵循 + 快速生成），三稿迭代非常合理。但对于本地 35B 模型，单稿 + 自检 + 去AI化已经是最优性价比。

2. **"理想工作流"和"可用工作流"应该分开定义**。Skill 文件可以保留完整的 7 步流程作为"标准模式"，但默认应该是 quick 模式，用户在确认基础效果满意后再选择升级。

3. **迭代策略应该是"修不是重写"**。当前的迭代是"重新生成三稿"，更高效的方式是"基于最佳稿的缺陷清单，做定向修改"——这只需要一次短 prompt 调用，而不是三次完整生成。

## 五、作家风格提取的改进方向：从规则驱动到数据驱动

### 当前方案的本质

当前 HECAITOU_STYLE_SKILLS.md 是一份**人工提炼的规则集**：由 AI 阅读 1063 篇文章后总结出来的风格特征、结构模板、禁忌清单。它的本质是"把隐性知识显性化"，然后希望模型通过遵守规则来模仿风格。

### 这个方案的天花板

1. **规则无法捕捉韵律**。和菜头的文字有一种特殊的节奏——长短句交替、破折号的使用时机、段落之间的呼吸感。这些是"读 100 篇你就能感觉到，但写 100 条规则你也说不清"的东西。规则只能告诉模型"短句为主"，但不能教会它"什么时候该用一个 37 字的长句来打破节奏"。

2. **规则互相冲突时模型无所适从**。"温和的刻薄"——什么程度算温和？什么程度算刻薄？"自嘲先行，但不矫情"——自嘲和矫情的边界在哪里？这些模糊地带恰恰是风格的精髓，但规则系统处理不了。

3. **规则膨胀不可避免**。从 v1.0 到 v4.0，规则从几百字膨胀到 13,528 字，因为每发现一个"模型做错的地方"就加一条规则。但规则越多，token 越紧张，模型反而越难遵守——这是一个负反馈循环。

### 未来改进路线

**短期（不改技术栈）**：
- **示例优于规则**。把"风格约束"章节从 20 条规则缩减为 5 条核心规则 + 3 段精选原文片段。模型通过模仿示例学到的风格远比遵守规则多。
- **动态 Skill 注入**。根据分类结果，从预处理好的"每类 3 段典型片段"中选取注入，而不是每次都加载完整 Skill。
- **建立量化评估基准**。随机抽 20 篇原文，让模型判断"这是和菜头写的还是 AI 写的"，建立一个 baseline 准确率。每次改 Skill 都跑一遍，看准确率是否提高。

**中期（升级检索）**：
- **BM25 替代关键词匹配**。用 `rank_bm25` 库替换当前的 jieba 分词 + 简单匹配，成本几乎为零，检索质量显著提升。
- **段落级检索**。当前是文章级检索，但一篇 2000 字的文章中真正体现风格的可能只有 2-3 个段落。段落级检索能在同样的 token 预算内提供更精准的风格参考。

**长期（改变模型）**：
- **LoRA 微调仍然是终极方案**。1063 篇 × 平均 1750 字 = 186 万字的训练数据在中文个人作者语料中极为充足。用 Qwen2.5-7B + LoRA 在 AutoDL 上花 ¥50 训练 4 小时，得到的风格还原度将远超任何 prompt engineering。
- **对比学习**：不只教模型"怎么写像和菜头"，还教它"怎么写不像和菜头"。用 DPO（Direct Preference Optimization），正例是原文，负例是 AI 的初始输出。

## 六、数据工程的投入产出比：爬虫阶段的经验（AI 额外提出的方向 ①）

### 发生了什么

Day 1 的前 30 分钟完成了爬虫开发，45 分钟内爬完了 1063 篇文章（14.6 MB 原始数据）。随后花了约 2 小时做分类（classify_articles.py），产出 6 个分类文件（40,037 行）。

这个阶段的 ROI 极高——**占用项目 ~15% 的时间，但提供了后续一切工作的基础**。

### 被忽略的数据质量问题

1. **HTML → Markdown 转换的损失**。markdownify 库在处理嵌套列表、引用块、图片 alt 文本时有已知问题。部分文章的格式可能已经损坏，但没有做过质量抽检。

2. **分类准确率未验证**。classify_articles.py 用的是纯关键词匹配（无 LLM），分类准确率未知。一篇文章可能同时被分到多个类别，也可能被分错类别。错误分类会导致检索时返回不相关的参考文章。

3. **重复/近似文章未去重**。和菜头有时会修改旧文重发，1063 篇中可能有少量近似重复。

### 可复制的经验

1. **爬虫阶段追加一个质量抽检步骤**。随机抽 20 篇，对比原始网页和 Markdown 输出，计算一个"还原率"。如果低于 95%，需要修复转换逻辑。

2. **分类结果需要人工抽样验证**。随机从每个类别抽 10 篇，人工判断分类是否正确，统计一个准确率。如果低于 80%，需要引入 LLM 辅助分类（哪怕是用 Ollama 本地模型做一遍验证）。

3. **数据工程的时间分配应该是 30% 采集 + 70% 清洗/验证**。本项目的实际分配是 80% 采集 + 20% 分类，清洗和验证几乎为零。这在 PoC 阶段可以接受，但如果要做 LoRA 微调，数据质量将直接决定模型质量。

## 七、AI 辅助编程的协作模式：人机分工的效率瓶颈（AI 额外提出的方向 ②）

### 本项目的协作模式

```
用户：提出需求 + 提供运行环境 + 反馈运行结果
AI  ：设计 + 编码 + 调试 + git 操作 + 文档
```

AI 承担了 ~95% 的编码工作，用户承担了 100% 的运行测试。这种分工模式的瓶颈在于**反馈延迟**：

- AI 写完代码 → 用户下载到本地 → 用户运行 → 用户截图/粘贴结果 → AI 分析
- 每个反馈循环大约需要 10-30 分钟（取决于用户响应速度和网络状况）
- 4 天内大约完成了 15-20 个反馈循环

### 效率瓶颈分析

1. **AI 无法运行目标环境的代码**。爬虫可以在沙箱中运行（但没有做），Ollama 相关代码完全无法在沙箱中测试。这意味着 AI 的所有修改都是"盲改"——只能通过阅读代码推断问题，而不能实际运行验证。

2. **Git 操作成为最大的时间浪费**。本项目在 git push / PR 创建 / PR 链接验证上花费了大量对话轮次。这些操作本身没有技术价值，但因为用户网络不稳定 + AI 对远程仓库状态判断不准确，反复出错。

3. **错误诊断严重依赖用户提供的信息质量**。用户粘贴完整日志时（如"Ollama 超时 600s"），AI 能快速定位问题。用户只说"效果不好"时，AI 需要额外几个来回才能缩小范围。

### 可复制的经验

1. **在项目启动时建立"诊断协议"**。给用户一个标准化的信息收集模板：

   ```
   报告模板：
   1. 运行的命令：___
   2. 完整错误输出（最后 50 行）：___
   3. 运行时间：___
   4. 输出文件内容（如有）：___
   5. 系统信息：内存___GB，CPU___，有无GPU___
   ```

2. **把 git 操作自动化到脚本中**。与其让 AI 每次手动执行 git 命令，不如在 Day 1 就创建一个 `deploy.sh`，用户只需要运行一条命令就能同步代码。

3. **优先让可以在沙箱运行的部分在沙箱运行**。爬虫阶段的代码完全可以在沙箱中开发和测试，不需要等用户反馈。但本项目中，AI 直接在沙箱中运行了爬虫并推送了数据——这个做法是对的。问题出在后续的 Ollama 相关代码，沙箱无法模拟。对于这种情况，应该写单元测试来覆盖非 Ollama 部分（如 prompt 构建、token 预算计算、文章解析），减少"盲改"的概率。

4. **AI 应该主动在代码中嵌入诊断工具**。最终版的 token 预算诊断输出应该从 v1.0 就存在。类似的，每个关键步骤都应该有 `--dry-run` 选项，让用户在不实际调用 Ollama 的情况下验证流程是否正确。

---

## 经验总结：7 条可复制的原则

| # | 原则 | 本项目的教训 |
|---|------|-------------|
| 1 | **Token 预算是第一性约束** | 系统提示从几百字膨胀到 13,528 字，无人计算，直到超时才发现 |
| 2 | **先确认运行环境，再选技术路径** | 写了 7 路径对比文档，最终受限于"本地 Ollama 无 GPU"，只能走最简方案 |
| 3 | **尽早让用户运行最小版本** | 如果 Day 1 就跑一次 `--quick` 测试，Day 3 的 token 溢出问题可能 Day 1 就能发现 |
| 4 | **工作流复杂度匹配模型能力** | 三稿迭代是好设计，但在 35B MoE 本地模型上 ROI 很低 |
| 5 | **示例优于规则** | 20 条风格规则不如 3 段精选原文片段 |
| 6 | **数据质量需要验证，不能假设** | 爬虫和分类都跑通了，但没有做准确率抽检 |
| 7 | **在代码中嵌入诊断工具，不要等到出问题** | Token 预算输出、`--dry-run`、环境检查都应该从 v1.0 就存在 |
