# 构建「和菜头风格写作 Agent」—— 技术路径对比

## 语料概况

| 指标 | 数据 |
|------|------|
| 文章总数 | 1,063 篇 |
| 中文字数 | **186 万字** |
| 估算 Token 数 | ~278 万 tokens |
| 时间跨度 | 2016-11 ~ 2026-03 |
| 平均每篇 | ~1,750 字 |

这个数据量级在中文个人博客中属于非常充足的语料。

---

## 路径总览

| # | 路径 | 核心思路 | 难度 | 成本 | 效果上限 |
|---|------|---------|------|------|---------|
| 1 | **System Prompt + Few-shot** | 精心编写提示词 + 塞几篇范文 | ★☆☆☆☆ | 几乎免费 | ★★★☆☆ |
| 2 | **RAG 检索增强生成** | 建向量库，写作时检索相似文章做参考 | ★★☆☆☆ | 低 | ★★★★☆ |
| 3 | **Long-context 全量灌入** | 利用超长上下文窗口直接放入大量原文 | ★☆☆☆☆ | 中（API费用） | ★★★☆☆ |
| 4 | **LoRA / QLoRA 微调** | 在开源模型上用你的语料做参数微调 | ★★★★☆ | 中~高 | ★★★★★ |
| 5 | **全量微调 (Full Fine-tuning)** | 全参数微调大模型 | ★★★★★ | 很高 | ★★★★★ |
| 6 | **OpenAI / Claude Fine-tuning API** | 用闭源模型的官方微调接口 | ★★★☆☆ | 中 | ★★★★☆ |
| 7 | **Multi-Agent 工作流** | 多个 Agent 协作：风格分析→大纲→写作→润色 | ★★★☆☆ | 低~中 | ★★★★☆ |

## 路径 1：System Prompt + Few-shot（最简单，立刻能用）

### 做法
写一个精心设计的 System Prompt，描述和菜头的写作风格特征，然后在对话中附上 2~3 篇原文作为范例。

### 实现

```python
SYSTEM_PROMPT = """
你是一位资深中文博客作者，你的笔名叫和菜头，博客叫「槽边往事」。
请严格模仿以下写作风格进行创作：

【语气特征】
- 冷幽默，经常用反讽和自嘲
- 看似随意闲聊，实则逻辑严密
- 喜欢从一个微小的日常切口（喝酒、养猫、买东西）引入深层思考
- 常用短句收尾，制造戛然而止的效果
- 偶尔使用粗俗但精准的比喻（"这是屁话"）

【结构特征】
- 不用小标题，全文一气呵成
- 开头从具体事件/场景起笔，不说教
- 中段层层推进，从现象到本质
- 结尾常一句话总结，点到即止，不煽情
- 段落之间有节奏感，长短交替

【内容特征】  
- 关注互联网、AI、人际关系、自我认知
- 喜欢用三段式列举然后逐一拆解
- 经常引用自己的生活经验作为论据
- 对"正确的废话"保持警惕和嘲讽
- 文末常有"禅定时刻"或题图说明

【禁忌】
- 绝不用"赋能""抓手""底层逻辑"等互联网黑话
- 不写鸡汤，如果写了会立刻自我解构
- 不表演深刻，真正的洞察藏在闲聊里
"""
```

### 优缺点
- ✅ **10 分钟就能用**，零成本
- ✅ 不需要任何技术基础
- ❌ 上下文窗口有限，放不了多少范文
- ❌ 风格模仿停留在表面，缺乏深层文字肌理
- ❌ 容易"串味"，几轮对话后回到模型原始语气

### 适合谁
想快速体验效果的人。作为其他方案的辅助手段。

## 路径 2：RAG 检索增强生成（推荐性价比最高方案）

### 做法
把 1,063 篇文章切分建成向量数据库。写作时根据主题检索最相关的 5~10 篇文章，拼入 Prompt 作为风格参考，让模型在"有据可依"的情况下模仿写作。

### 架构

```
用户输入主题 → 向量检索 Top-K 相关文章 → 拼接 System Prompt + 检索结果 → LLM 生成
                    ↑
              向量数据库
           (1,063 篇文章 embeddings)
```

### 技术选型
- **向量数据库**：ChromaDB（本地轻量）/ Pinecone / Weaviate
- **Embedding 模型**：`text-embedding-3-small` 或 `bge-large-zh`
- **生成模型**：GPT-4o / Claude 3.5 Sonnet / DeepSeek
- **框架**：LangChain / LlamaIndex

### 核心代码思路

```python
# 1. 建库：把每篇文章向量化存入
for article in all_articles:
    chunks = split_article(article)  # 按段落切分
    embeddings = embed(chunks)
    vector_db.add(chunks, embeddings, metadata={date, title, url})

# 2. 写作时：检索 + 生成
def write_like_hecaitou(topic):
    relevant = vector_db.query(topic, top_k=8)  # 检索相似文章
    prompt = SYSTEM_PROMPT + "\n\n以下是你过去写过的相关文章供参考：\n"
    for doc in relevant:
        prompt += f"\n---\n{doc.text}\n"
    prompt += f"\n\n现在请以同样的风格写一篇关于「{topic}」的文章。"
    return llm.generate(prompt)
```

### 优缺点
- ✅ **效果好**：每次写作都有真实原文做参考，风格还原度高
- ✅ **成本低**：只需要 Embedding 一次 + 每次调用 API
- ✅ **可解释**：能看到模型参考了哪些文章
- ✅ **易维护**：新文章直接加入向量库
- ❌ 需要搭建向量数据库（但 ChromaDB 很简单）
- ❌ 检索质量影响最终效果
- ❌ 模型核心能力仍是通用的，只是"参考着写"

### 适合谁
**大多数人的最佳选择**。平衡了效果、成本和工程复杂度。

## 路径 3：Long-context 全量灌入

### 做法
利用 Gemini 1.5 Pro（200万 token）/ Claude（20万 token）/ GPT-4o（12.8万 token）等超长上下文模型，直接把大量原文塞进上下文窗口。

### 实现
```python
# 把尽可能多的文章直接拼接放入
all_text = load_all_articles()  # 186万字 ≈ 278万 token

# Gemini 1.5 Pro 可以放全部！
# Claude 只能放约 13万字（约 80 篇）
# GPT-4o 只能放约 8.5万字（约 50 篇）

prompt = f"""
以下是和菜头「槽边往事」博客的全部文章：

{all_text}

---
请严格模仿以上文章的语气、结构和视角，写一篇关于「{topic}」的文章。
"""
```

### 优缺点
- ✅ **极其简单**，无需任何工程
- ✅ Gemini 1.5 Pro 理论上可以放入全部 186 万字
- ❌ **非常贵**：每次调用要处理几百万 token
- ❌ "大海捞针"问题——太多内容模型反而抓不住重点
- ❌ 上下文越长，风格一致性反而可能下降
- ❌ 延迟高（几十秒到几分钟）

### 适合谁
想要一次性测试效果、不想搭建任何系统的人。建议用 Gemini 1.5 Pro 测试。

## 路径 4：LoRA / QLoRA 微调（效果天花板最高）

### 做法
选一个中文基座模型（如 Qwen2.5-7B/14B、DeepSeek-V2、Yi-1.5），用 LoRA 技术在和菜头语料上做低秩微调。模型会从参数层面学到写作风格。

### 数据准备

```jsonl
{"instruction": "请以和菜头的风格写一篇关于宿醉后情绪变化的文章", "output": "昨天去朋友家喝了两杯...（原文）"}
{"instruction": "请以和菜头的风格写一篇关于失恋的文章", "output": "人在失恋之后，通常会说三句话...（原文）"}
```

需要为每篇文章构造 instruction（可用 GPT 批量生成），大约 1,063 条训练数据。

### 技术要求

| 项目 | 要求 |
|------|------|
| GPU | 单卡 24GB（如 RTX 4090）可跑 7B LoRA |
| | 单卡 80GB（如 A100）可跑 14B~72B |
| 框架 | LLaMA-Factory / Unsloth / Axolotl |
| 训练时间 | 7B 模型约 2~4 小时（单 4090） |
| 基座模型 | Qwen2.5-7B-Instruct（推荐）|
| | DeepSeek-V2-Lite |

### 训练示例（用 LLaMA-Factory）

```yaml
# train_config.yaml
model_name_or_path: Qwen/Qwen2.5-7B-Instruct
dataset: hecaitou_articles
finetuning_type: lora
lora_rank: 64
lora_alpha: 128
learning_rate: 2e-4
num_train_epochs: 3
per_device_train_batch_size: 4
gradient_accumulation_steps: 4
bf16: true
output_dir: ./hecaitou-lora
```

### 优缺点
- ✅ **效果最好**：模型从参数层面学会了风格
- ✅ 一次微调，永久使用，推理成本低
- ✅ LoRA 适配器很小（几十 MB），可分享
- ✅ 186 万字 / 1063 篇是非常充足的微调语料
- ❌ **需要 GPU**（云 GPU 也行，AutoDL/Vast.ai 约 ¥5~15/小时）
- ❌ 需要一定的机器学习知识
- ❌ 数据准备（构造 instruction-output 对）需要时间
- ❌ 调参需要经验（学习率、epoch数、LoRA rank）

### 适合谁
有一定技术背景、追求最佳效果的人。这是让 AI **真正"变成"和菜头** 的唯一路径。

---

## 路径 5：全量微调（不推荐）

### 做法
对大模型全部参数进行微调。

### 为什么不推荐
- 186 万字对全量微调来说**数据量太少**，容易过拟合
- 需要多卡高端 GPU（8×A100 级别），成本极高
- LoRA 在这个数据量级下效果几乎一样好
- 唯一优势是理论上限略高，但投入产出比极差

### 适合谁
不适合任何个人用户。除非你是 AI 公司且这是核心业务。

---

## 路径 6：OpenAI / Claude Fine-tuning API（省心的微调方案）

### 做法
用 OpenAI 的 fine-tuning API 微调 `gpt-4o-mini` 或 `gpt-3.5-turbo`。

### 数据准备
```jsonl
{"messages": [
  {"role": "system", "content": "你是和菜头，槽边往事的作者。"},
  {"role": "user", "content": "写一篇关于AI隧道视野的文章"},
  {"role": "assistant", "content": "我认为 AI 在飞快地批量制造人们的隧道视野..."}
]}
```

### 成本估算（OpenAI）
| 项目 | 费用 |
|------|------|
| 训练 (gpt-4o-mini) | ~$25（278万 token × $3/1M × 3 epochs） |
| 训练 (gpt-4o) | ~$75 |
| 推理 | 比基础模型贵 ~1.5 倍 |

### 优缺点
- ✅ **不需要 GPU**，全在云端完成
- ✅ 数据格式简单，上传即训
- ✅ 部署零成本，直接通过 API 调用
- ❌ 只能微调 OpenAI 提供的模型（选择有限）
- ❌ 模型权重不属于你，依赖 OpenAI 平台
- ❌ 微调深度有限，不如 LoRA 灵活
- ❌ 持续调用有费用

### 适合谁
不想碰 GPU 和训练环境、愿意付费换省心的人。

## 路径 7：Multi-Agent 工作流（工程化方案）

### 做法
不靠单一模型，而是设计一个多步骤工作流，用多个 Agent 分工协作。

### 架构

```
用户给出主题
    │
    ▼
┌─────────────────┐
│ Agent 1: 风格分析  │ ← 从向量库取 5 篇相关文章，提取风格要素
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Agent 2: 选题角度  │ ← 分析和菜头会从什么切口写这个话题
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Agent 3: 初稿生成  │ ← 带着风格要素 + 角度 + 参考文章写初稿
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Agent 4: 风格校对  │ ← 对比原文，修正语气/结构/用词偏差
└────────┬────────┘
         │
         ▼
       最终输出
```

### 优缺点
- ✅ 每个环节可独立优化
- ✅ 可组合使用 RAG + Prompt Engineering
- ✅ 最终效果可以很稳定
- ❌ 工程复杂度高，需要多次 API 调用
- ❌ 延迟较高（串行多步）
- ❌ 需要持续迭代 prompt 和流程

### 适合谁
想要做成产品级应用的开发者。

---

## 我的推荐

### 如果你是普通用户（非技术背景）

> **路径 1 + 路径 3**
>
> 先用 System Prompt 快速体验，再用 Gemini 1.5 Pro 灌入全部文章测试。
> 总花费：接近 0。
> 所需时间：30 分钟。

### 如果你是开发者（会写代码但没有 GPU）

> **路径 2（RAG）+ 路径 6（OpenAI 微调）**
>
> 先建一个 RAG 系统日常使用，同时花 $25 用 OpenAI 微调一个专用模型。
> 两条路各自独立，可以对比效果。
> 总花费：~$30。
> 所需时间：1~2 天。

### 如果你追求极致效果

> **路径 4（LoRA 微调）+ 路径 2（RAG）组合**
>
> 用 Qwen2.5-7B 做 LoRA 微调得到风格化模型，再套一层 RAG 在推理时补充参考文章。
> 这是效果天花板最高的方案。
> 总花费：云 GPU ~¥50-100 + 时间。
> 所需时间：3~5 天（含数据准备和调参）。
